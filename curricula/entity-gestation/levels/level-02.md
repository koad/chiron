# Level 2: Configuring `.env` — Identity, Runtime, and the MOTHER Relationship

## Learning Objective

After completing this level, the operator will be able to:
> Expand the minimal `.env` skeleton that gestation produces into a complete entity identity declaration, explain what each field controls at runtime, and confirm whose name will appear on commits before any git operation runs.

**Why this matters:** The gestation script writes a minimal `.env` — just enough to bind the entity's name and bind address. Everything else — git authorship, harness selection, home directory paths, ENTITY_HOST — must be added before the entity first commits or first runs. An entity that runs before `.env` is complete will produce misattributed commits and may run from the wrong directory or on the wrong machine.

---

## Knowledge Atoms

## Atom 2.1: What Gestation Actually Writes — The Minimal .env

The `.env` written by the gestation script contains exactly two lines of content:

```env
# auto generated by koad:io entity gestate script
KOAD_IO_BIND_IP=127.0.0.1
METEOR_PACKAGE_DIRS=/home/koad/.koad-io/packages
```

`KOAD_IO_BIND_IP` sets the IP address the entity binds to for local services. `127.0.0.1` is the loopback address — the entity is not reachable from the network until this is changed. This is a safe default: a freshly gestated entity should not be network-accessible until the operator deliberately opens it.

`METEOR_PACKAGE_DIRS` points to the framework's package directory at `~/.koad-io/packages`. This is the path the framework uses to find packages it loads at runtime. It is set using the absolute path resolved at gestation time, not a variable reference, which means the path is correct only if the entity runs on the same machine where it was gestated. If the entity is cloned to a different machine, this path must be updated.

Everything else — the entity's name, its git identity, its home directory, the AI harness it should use, its role and purpose — is absent. The gestation script does not write these fields because they require operator intent that cannot be inferred. The name of the entity as a proper noun (for `GIT_AUTHOR_NAME`) differs from its lowercase path identifier. The email address is not predictable. The harness choice is a deliberate decision. These must be filled in by the operator.

**What you can do now:**
- Read `~/.juno/.env` and identify which two lines were written by gestation vs. everything else added afterward
- Explain why `METEOR_PACKAGE_DIRS` is an absolute path rather than a variable reference, and what must change if the entity moves to a different machine
- State why the gestation script deliberately leaves `GIT_AUTHOR_NAME` out of the initial `.env`

**Exit criterion for this atom:** The operator can recite the two gestation-written fields from memory, explain what each controls, and state what is deliberately absent and why.

## Atom 2.2: The Core Identity Fields — Who the Entity Is

The identity section of `.env` declares the entity to the framework and to git. Looking at Juno's `.env` as the reference implementation, the identity fields fall into three groups:

**Entity location:**
```env
ENTITY=juno
ENTITY_DIR=/home/koad/.juno
ENTITY_HOME=/home/koad/.juno/home/juno
```

`ENTITY` is the entity's canonical name. It must match the name used at gestation exactly (lowercase). This is what the framework reads to determine which entity is active when commands run inside the entity directory.

`ENTITY_DIR` is the absolute path to the entity's root directory. The framework uses this to resolve relative paths and to find the entity's commands, hooks, and configuration. Use the absolute path — no `~` expansion, no variables.

`ENTITY_HOME` is the entity's personal home directory inside its entity directory. For Juno, this is `~/.juno/home/juno/`. This is the entity's `$HOME` equivalent — where it writes files that belong to its personal space rather than its configuration.

**Git authorship:**
```env
GIT_AUTHOR_NAME=Juno
GIT_AUTHOR_EMAIL=juno@kingofalldata.com
GIT_COMMITTER_NAME=Juno
GIT_COMMITTER_EMAIL=juno@kingofalldata.com
```

These four fields determine whose name and email appear on every commit the entity makes. They are not optional and they are not defaulted from system git config — the framework reads these environment variables directly. `GIT_AUTHOR_NAME` and `GIT_COMMITTER_NAME` can differ (author is who wrote the code, committer is who made the commit), but for a single-entity context they are typically the same. The email follows the convention `entityname@kingofalldata.com` for koad:io team entities.

**What you can do now:**
- Read `~/.juno/.env` and trace each identity field to where it is used by the framework
- Write the six identity fields (ENTITY, ENTITY_DIR, ENTITY_HOME, plus the four git fields) for a hypothetical entity named `nova` operating at `/home/koad/.nova/`
- Confirm that `git log --pretty=format:"%an %ae" -1` in `~/.juno/` shows "Juno juno@kingofalldata.com"

**Exit criterion for this atom:** The operator can write the complete identity section of `.env` for a new entity without consulting a reference, and explain what each field controls at runtime.

## Atom 2.3: Runtime and Operational Fields — Where the Entity Runs and How

Beyond identity, `.env` carries fields that configure how the entity operates at runtime. These are not required for the entity to commit and push, but they are required for the entity to function correctly as an operational AI agent.

**`KOAD_IO_ENTITY_HARNESS`** — The AI harness the entity uses. Current values are `claude` (Claude Code) and `opencode` (the framework default for entities without an explicit harness set). Team entities at koad:io — Juno, Vulcan, Veritas, Mercury, Muse — all set `KOAD_IO_ENTITY_HARNESS=claude`. Entities that do not set this field fall back to the framework default, which is currently `opencode`. Set this explicitly for any entity that should run under a specific harness.

**`KOAD_IO_QUIET=1`** — Suppresses the animated banners and spinners in the gestation command and other framework output. When set to `1`, the framework runs in non-interactive mode — output is clean text with no cursor animations. Required for entities that will be invoked by automated scripts or from within other processes. Juno sets `KOAD_IO_QUIET=1` because it is frequently invoked from Claude Code sessions where terminal animations do not render correctly.

**`ENTITY_HOST`** — The machine the entity is intended to run on. This is informational and used for routing in the daemon and for display in operations boards. Set it to the machine's hostname where the entity will primarily operate. For entities on thinker, this is `thinker`.

**Public identity fields** (Juno's extended configuration):
```env
ENTITY_KEYS=/home/koad/.juno/juno.keys
TRUST_CHAIN=/home/koad/.juno/trust
CREATOR=koad
CREATOR_KEYS=canon.koad.sh/koad.keys
ROLE=business-operations
PURPOSE="Operate the business side of koad:io ecosystem"
```

These fields are not read by the framework at startup — they are documentation within the `.env` file for operators and other entities reading it. `CREATOR` and `CREATOR_KEYS` record who established the entity and where their public key can be verified. `ROLE` and `PURPOSE` are human-readable identity declarations that other entities can read to understand the new entity's function.

**What you can do now:**
- Read `~/.juno/.env` and identify which fields are read by the framework vs. which are documentary
- Set `KOAD_IO_ENTITY_HARNESS` for a hypothetical entity that should run under Claude Code
- Explain what would happen to an entity's terminal output if `KOAD_IO_QUIET` were not set (animated spinners would appear, which may corrupt output in Claude Code sessions)

**Exit criterion for this atom:** The operator can distinguish framework-read fields from documentary fields, set the harness field correctly, and explain when `KOAD_IO_QUIET=1` is necessary.

## Atom 2.4: The `.credentials` File — Secrets That Never Enter Git

Every entity has two files for environment configuration: `.env` and `.credentials`. They are not interchangeable. Understanding the distinction is critical before the first `git add`.

**`.env`** is committed to git. It contains configuration — values that describe the entity's identity and runtime preferences but that contain no secrets. API keys, tokens, and passwords never go in `.env`.

**`.credentials`** is gitignored. It contains secrets — values that, if exposed in a public git repository, would allow unauthorized parties to act as the entity. Typical contents:

```env
ANTHROPIC_API_KEY=sk-ant-...
GITHUB_TOKEN=ghp_...
```

The `.credentials` file is sourced by the framework alongside `.env`. From the runtime's perspective, values in `.credentials` are available in the environment just as values in `.env` are. The difference is entirely about what gets committed.

Check `.gitignore` before the first `git add` and verify that `.credentials` is covered. Juno's `.gitignore` covers it via the `credentials` and `*.credentials` patterns. If you have modified the `.gitignore` or are working from a freshly gestated entity whose `.gitignore` may use different patterns, run:

```bash
git check-ignore -v .credentials
```

If this command returns nothing, `.credentials` is not ignored and you must add it to `.gitignore` before adding any files to the staging area.

The cost of a credentials leak is not recoverable through git history rewriting. GitHub scans all pushes for exposed secrets and revokes them automatically in some cases, but not all. The correct approach is preventive: confirm `.credentials` is gitignored before any `git add`, every time, on every entity.

**What you can do now:**
- Run `git check-ignore -v ~/.juno/.credentials` and confirm it is covered (even though the file likely doesn't exist — the ignore pattern applies regardless)
- State the two values that typically live in `.credentials` for a koad:io entity
- Explain the one-sentence rule for `.env` vs. `.credentials`: configuration goes in `.env`, secrets go in `.credentials`

**Exit criterion for this atom:** The operator can state the distinction between `.env` and `.credentials` in one sentence, verify that `.credentials` is gitignored before committing, and name the two secrets that typically live there.

## Atom 2.5: The Environment Cascade — Which Value Wins

The framework loads environment values from multiple sources, and understanding the cascade order is necessary for debugging when a value is not what you expect.

The cascade, from lowest to highest priority:

1. **Framework defaults** — Values the koad:io framework assumes if nothing else sets them. These are thin: the framework does not assume entity names or git identity.
2. **Framework `.env`** (`~/.koad-io/.env`) — Framework-wide configuration that applies across all entities. If the framework has an `ANTHROPIC_API_KEY` here, all entities inherit it unless they override.
3. **Entity `.env`** (`~/.<entity>/.env`) — Entity-specific configuration. Overrides the framework `.env` for fields set in both.
4. **Entity `.credentials`** (`~/.<entity>/.credentials`) — Entity secrets. These override both `.env` files for the same field names.
5. **Shell environment** — Values already set in the shell when the entity is invoked. These override everything below them in this list.

In practice, the cascade means: if you set `ANTHROPIC_API_KEY` in both the framework `.env` and the entity `.credentials`, the entity `.credentials` value wins. If you export `ANTHROPIC_API_KEY` in your shell before invoking the entity, the shell value wins over everything.

The cascade also matters for git identity. If the system git config at `~/.gitconfig` sets a `user.email`, but the entity's `.env` sets `GIT_COMMITTER_EMAIL`, the environment variable wins over git config. This is the mechanism that makes per-entity git identity work: the entity's `.env` sets `GIT_AUTHOR_NAME` and `GIT_COMMITTER_NAME` as environment variables, which git reads at higher priority than its own config file.

To debug a value at runtime:

```bash
# Load the entity environment and echo the value
source ~/.entityname/.env && echo $GIT_AUTHOR_NAME
```

Or check what git actually sees:

```bash
cd ~/.entityname && git var GIT_AUTHOR_IDENT
```

The `git var` command shows the full author identity string git will use for the next commit, incorporating all environment and config sources.

**What you can do now:**
- Run `cd ~/.juno && git var GIT_AUTHOR_IDENT` and confirm it shows the correct author identity
- Explain what would happen if you exported `GIT_AUTHOR_NAME=wrong-name` in your shell before committing from `~/.juno/`
- Describe where to look first when an entity's commits show the wrong author name (check shell environment first, then entity `.env`, then framework `.env`, then system git config)

**Exit criterion for this atom:** The operator can state the five-level cascade from memory, lowest to highest priority, and use `git var GIT_AUTHOR_IDENT` to verify the identity that will appear on the next commit.

## Assessment

**Question:** "A colleague's entity `argus` is making commits attributed to 'koad koad@example.com' instead of 'Argus argus@kingofalldata.com'. What are the three most likely causes, in order of likelihood?"

**Acceptable answers:**
1. `GIT_AUTHOR_NAME` and `GIT_AUTHOR_EMAIL` are missing from `~/.argus/.env`
2. The shell has `GIT_AUTHOR_NAME` exported from a previous context (shell wins over `.env`)
3. The entity `.env` is not being sourced before the commit runs (the framework environment loader is not running)

**Red flag answers (indicates level should be revisited):**
- "Edit `~/.gitconfig` to fix the name" — git config is overridden by environment variables; editing it will not fix an environment variable problem
- "Rerun gestation" — `.env` configuration errors do not require regeration of keys or directories

## Alice's Delivery Notes

The `.env` vs. `.credentials` distinction is the lesson that prevents the most consequential mistake in this curriculum. Spend time on it. The operator has likely worked with `.env` files in software projects before and may have habits around what goes there. Emphasize the single rule: if it is a secret (API key, token, password), it goes in `.credentials`.

The `git var GIT_AUTHOR_IDENT` command is a gift — it is the definitive answer to "what identity will appear on my next commit?" Make it a habit before Level 3's git init work begins. Operators who internalize this check will never have misattributed commits.

The cascade section is the most complex atom in this level. Resist the temptation to drill the full five levels — instead, focus on the two most common debugging scenarios: shell environment overriding `.env` (source of most misattribution bugs), and entity `.credentials` overriding framework `.env` (expected behavior for per-entity API keys). Those two scenarios cover 95% of the debugging the operator will encounter.

Do not cover passenger.json in this level — that is Level 6. Do not cover trust bonds — that is Level 5. This level ends at a correctly configured `.env`.
