# Level 3: The Passengers Collection — Reading Live State

## Learning Objective

After completing this level, the operator will be able to:
> Open the admin PWA at localhost:9568, read the Passengers collection, identify the currently selected entity from the `selected` field, explain what the `all` and `current` publications expose, and understand the known gap: no liveness signal means all registered entities appear alive indefinitely.

**Why this matters:** The Passengers collection is the daemon's view of the entity roster. An operator who cannot read it cannot tell whether an entity is registered, which entity is currently selected in the browser extension, or whether a registration change took effect. Reading this state is a basic operational skill.

---

## Knowledge Atoms

### Atom 3.1: The Admin PWA at localhost:9568

The admin PWA is the primary interface for reading live daemon state. It is a browser-based application served by the daemon's HTTP server. Open it by navigating to `http://localhost:9568` in any browser on the same machine. No login, no authentication — the daemon binds to localhost only (by default), so access from the same machine is unrestricted.

What you see at `localhost:9568`:

**The Passengers list.** Every registered entity appears here with its display name, role, avatar, and outfit color. The list reflects the current state of the Passengers collection in MongoDB — it updates reactively when the collection changes.

**The selected entity indicator.** One entity in the list is marked as currently selected. This is the entity the Dark Passenger browser extension is currently paired with. The selection indicator changes in real time when `passenger.check.in` is called.

**The Reload button.** Triggers `passenger.reload` — re-scans entity directories and repopulates the collection. This is the most useful operational control in the admin PWA.

**The quick-launch buttons.** For each registered entity, the buttons defined in that entity's `passenger.json` appear. Clicking a button from the admin PWA has the same effect as clicking from the browser overlay.

**What the admin PWA looks like when things are wrong:**

- If the daemon is not running: `localhost:9568` returns `Connection refused`. The admin PWA cannot load.
- If MongoDB is not connected: the PWA loads but shows an empty Passengers list. The reload button triggers but produces no entries.
- If no entities are registered: the PWA shows an empty Passengers list with no error — this is normal for a fresh install before any `passenger.json` files are added.

An empty list in the admin PWA has three possible causes: daemon not running (can't load the PWA at all), MongoDB disconnected (PWA loads but list is empty and reload fails), or no entity has a valid `passenger.json` (PWA loads, reload runs, list stays empty). The health endpoint distinguishes the first two; the startup log distinguishes the third.

**What you can do now:**
- Open the admin PWA at `localhost:9568` and identify each section
- Read the Passengers list and identify the currently selected entity
- Describe what an empty Passengers list could mean and how to determine which cause applies

**Exit criterion for this atom:** The operator can open the admin PWA, read the Passengers list, and explain what they are seeing — including what an empty list means and what to check next.

### Atom 3.2: The Passengers Collection Schema

Every registered entity produces one MongoDB document in the Passengers collection. Reading this document directly — rather than through the PWA — gives full visibility into the raw state the daemon is holding.

A complete Passengers document for a registered entity looks like this:

```json
{
  "_id": "aBcDeFgHiJkLmNoP",
  "handle": "juno",
  "name": "Juno",
  "image": "data:image/png;base64,iVBORw0KGgoAAAANS...",
  "outfit": { "hue": 72, "saturation": 45, "brightness": 28 },
  "buttons": [
    { "label": "Status", "action": "status", "description": "Current ops status" },
    { "label": "Inbox", "action": "inbox", "description": "Check comms inbox" },
    { "label": "Issues", "action": "issues", "description": "Open GitHub issues" }
  ],
  "selected": "2026-04-05T14:22:07.000Z"
}
```

Each field's origin:

**`_id`** — MongoDB's internal document identifier. Generated by MongoDB at insert time. Stable across reloads only if the collection is not cleared — `passenger.reload` removes all documents and re-inserts them, so `_id` values change on every reload.

**`handle`** — copied directly from the `handle` field in `passenger.json`. Used as the lookup key for registration checks (preventing duplicates) and as the entity identifier in operational contexts.

**`name`** — copied from the `name` field in `passenger.json`. This is the display name shown in the PWA and browser overlay. It is also the lookup key for `passenger.check.in(passengerName)` — the method searches the collection for `{ name: passengerName }`.

**`image`** — the base64-embedded avatar data URI. Generated by the daemon at registration time from the `avatar` file. Not the file path — the embedded data. Clients do not need filesystem access to display the avatar.

**`outfit`** — the HSB color object. Either copied from `passenger.json` or generated deterministically from the handle hash. Always present; never null.

**`buttons`** — the array copied from `passenger.json`. May be empty (`[]`) if no buttons were specified.

**`selected`** — this field exists only on the currently selected entity. When an entity is checked in via `passenger.check.in`, the daemon sets `selected: new Date()` on that entity and removes the `selected` field from all others. An entity document without a `selected` field is not currently selected. An entity document with `selected` set to a timestamp is the current entity. Only one document can have this field at a time.

**What you can do now:**
- Read a Passengers collection document (via the admin PWA's developer tools or any MongoDB client) and identify what each field came from
- Explain the difference between a document with `selected` and one without
- State why `_id` values change after a `passenger.reload`

**Exit criterion for this atom:** The operator can read a Passengers document and trace each field back to its source — `passenger.json` fields, daemon-generated fields, and the `selected` timestamp.

### Atom 3.3: The `current` and `all` Publications

The Passengers collection is exposed to clients (browser extension, desktop widget, admin PWA) via two DDP publications. Understanding what each publication exposes is important for understanding what clients see in different states.

**The `all` publication:**

```js
Meteor.publish('all', function() {
  return Passengers.find();
});
```

`all` returns the entire Passengers collection with no filter. Every registered entity, whether selected or not. This is what the admin PWA subscribes to for its full roster view. Subscribers to `all` see every entity the daemon has registered and receive real-time updates when the collection changes (reload, check-in, etc.).

**The `current` publication:**

```js
Meteor.publish('current', function() {
  return Passengers.find({ selected: { $exists: true } }, { sort: { selected: 1 } });
});
```

`current` returns only the document(s) where the `selected` field exists. Because the check-in mechanism ensures only one document has `selected` at a time, this publication delivers at most one document — the currently selected entity. This is what the desktop widget subscribes to: it shows the one entity currently active, and nothing when no entity is selected.

**Practical implications:**

If no entity has been checked in, subscribing to `current` returns zero documents. The widget shows nothing. This is not an error — it is the "no entity selected" state.

If an entity has been checked in, subscribing to `current` returns exactly one document. The widget displays that entity's name, avatar, outfit, and buttons.

The admin PWA subscribes to `all` for its roster view and also subscribes to `current` to highlight the selected entity in the list. Both subscriptions are active simultaneously — changes to either update the display in real time.

**What the publications do NOT expose:**

Neither publication exposes entity session state, git history, GitHub issues, trust bonds, or any information from outside the Passengers collection. The publications are a narrow window onto one MongoDB collection — the entity roster as the daemon knows it. An entity's full operational context lives in its entity directory and issue tracker, not in the daemon.

**What you can do now:**
- Explain the difference between what `all` and `current` return
- State what subscribing to `current` returns when no entity is checked in
- Explain why a widget that subscribes to `current` shows nothing when an entity is not selected

**Exit criterion for this atom:** The operator can explain both publications by name, describe exactly what each returns in the checked-in and not-checked-in states, and state what is outside the scope of both.

### Atom 3.4: Selecting an Entity — passenger.check.in

`passenger.check.in(passengerName)` is the DDP method that changes which entity is active. It is called by the Dark Passenger browser extension when the operator selects a different entity from the overlay, and by the admin PWA when an entity is clicked.

The method's behavior:

```js
'passenger.check.in'(passengerName) {
  // Find the entity by display name
  const passenger = Passengers.findOne({ name: passengerName });
  if (!passenger) return Meteor.Error('not-found', 'Passenger not found');

  // Remove 'selected' from all documents
  Passengers.update({}, { $unset: { selected: '' } }, { multi: true });

  // Set 'selected' on the target entity
  Passengers.update(passenger._id, { $set: { selected: new Date() } });

  return { _id: passenger._id };
}
```

Three things happen in sequence: lookup by name, deselect all, select the target.

**The name-based lookup has an important implication:** the argument to `passenger.check.in` is the display name (`"Juno"`, `"Vesta"`) — not the handle (`"juno"`, `"vesta"`). Case matters. If the entity's `name` field in `passenger.json` is `"Juno"` and you call `passenger.check.in("juno")`, the lookup fails and returns a `not-found` error.

**The timestamp records when selection changed:**

The `selected` field is set to `new Date()` — the current timestamp at time of check-in. This means you can read the `selected` timestamp from the collection document to know when the operator last switched to this entity. The `current` publication sorts by `{ selected: 1 }` — ascending by check-in time — which is only relevant if, due to a bug, multiple documents somehow had `selected` set simultaneously (the `multi: true` unset should prevent this, but the sort provides a deterministic tiebreaker).

**What you can do now:**
- Call `passenger.check.in` via the admin PWA and verify the selected indicator changes in real time
- Explain what argument `passenger.check.in` takes (display name, not handle) and why case matters
- Read the `selected` timestamp from a collection document and interpret what it means

**Exit criterion for this atom:** The operator can select an entity via the admin PWA, observe the `selected` field change in the collection, and explain the method's three-step behavior.

### Atom 3.5: The Liveness Gap

The Passengers collection shows which entities are registered. It does not show which entities are currently running an active session. This is a known architectural gap, and operators who miss it will misread the collection.

Here is the gap precisely:

When an entity registers at daemon startup, its document is inserted into the Passengers collection. When the daemon is reloaded, all documents are cleared and re-inserted for every entity with a valid `passenger.json`. Neither action — registration nor reload — has any connection to whether an entity's Claude Code session, opencode process, or other harness is currently running.

A `passenger.json` file that existed three days ago registers the entity today. An entity whose session ended six hours ago is still in the Passengers collection. The `selected` timestamp tells you when the entity was last checked in — not when its session last had activity.

There is no heartbeat. The daemon does not ping entities, entities do not ping the daemon, and there is no periodic mechanism that removes stale entries. An entity appears in the `all` publication indefinitely after registration — until `passenger.reload` is called and the entity's `passenger.json` is still present in its directory.

**Practical implications:**

- Seeing an entity in the Passengers collection does not mean the entity has an active session.
- Checking in an entity (`passenger.check.in`) does not start or wake the entity — it only updates the collection document.
- URL context delivered via `passenger.ingest.url` goes to whichever entity is selected, regardless of whether that entity's session is running. If no session is running to receive it, the URL context is lost.
- The admin PWA's entity list is a registration roster, not a live activity monitor.

**This is Sibyl's architectural hardening recommendation #1:** add a liveness mechanism — a periodic heartbeat that entities send to the daemon, with documents removed or marked stale when no heartbeat arrives within a defined window. As of 2026-04-05, this is not implemented. Operators who need to know whether an entity is live must check by other means: `ps aux | grep claude`, checking the entity's recent git commits, or reading the entity's terminal output.

The liveness gap is not a bug to work around — it is a design limitation to understand. An operator who understands it will not rely on the Passengers collection to determine session status. An operator who misses it will make incorrect inferences about entity availability.

**What you can do now:**
- Explain in one sentence what the Passengers collection shows and what it does not show
- Describe how to determine whether an entity actually has an active session (since the collection cannot tell you)
- State Sibyl's architectural recommendation and why it addresses the gap

**Exit criterion for this atom:** The operator can explain the liveness gap clearly — what the collection shows, what it does not show, and why presence in the collection is not evidence of an active session.

## Exit Criterion

The operator can open the admin PWA, identify the currently selected entity, read a Passengers collection document and name what each field means, explain what the `all` and `current` publications return, and explain why an entity appearing in the collection does not prove it is currently running.

**Verification question:** "You open the admin PWA and see six entities in the Passengers list, including Vulcan. Vulcan's `selected` timestamp is from yesterday. Is Vulcan currently running?"

Expected answer: Unknown — and probably not, but the collection cannot tell us. The `selected` timestamp shows when Vulcan was last checked in. The presence in the collection shows that Vulcan has a `passenger.json` in their entity directory and the daemon registered it. Neither of these facts indicates whether Vulcan's session is currently active. To determine that, check via `ps aux | grep claude` or by looking at Vulcan's recent git activity.

## Assessment

**Question 1:** "What does subscribing to `current` return when no entity has been checked in?"

**Acceptable answers:**
- "Zero documents. `current` publishes `{ selected: { $exists: true } }`. If no entity has the `selected` field, the subscription returns an empty cursor."
- "Nothing — an empty result set. That's the correct state when no entity is selected."

**Red flag answers:**
- "An error" — no error is returned; an empty cursor is valid and expected
- "All entities" — that's the `all` publication; `current` is filtered to selected only

**Question 2:** "What is the liveness gap and why does it matter?"

**Acceptable answers:**
- "There's no heartbeat mechanism. Entities stay in the Passengers collection indefinitely after registration, regardless of whether they're actually running a session. An operator who reads presence in the collection as 'entity is live' will be wrong. You have to check actual process state separately."
- "The daemon has no way to know if an entity's session is actually active. Registration is permanent until reload. Sibyl's recommendation is to add a heartbeat, but it's not implemented yet."

## Alice's Delivery Notes

Operators at this level are working with live state for the first time. They have registered entities (Level 2) and now need to read what the daemon is holding. Keep the admin PWA open throughout this level — every atom has something visible in the PWA.

The collection schema atom (3.2) is denser than the others. The key insight to deliver: every field in the document traces back to exactly one source — either `passenger.json`, the daemon's own generation logic, or the check-in method. Tracing those origins gives operators a mental model they can use to debug: "this field is wrong" → "where does this field come from?" → fix that source.

The `current` vs. `all` distinction in Atom 3.3 is where operators need to understand DDP publications concretely. Do not rely on "they just know what DDP is" from the prerequisite. Frame it as: "two views of the same collection, filtered differently." `all` = everything; `current` = just the selected one.

Atom 3.5 (the liveness gap) is this level's most important conceptual delivery. It is the correction to an assumption operators will almost certainly bring: "if it's in the roster, it's running." State the gap explicitly and bluntly. The recommended framing: "the Passengers collection is a registration book, not a heartbeat monitor."

The assessment question about checking whether Vulcan is running is designed to surface operators who have not fully absorbed the liveness gap. A correct answer names two alternative verification methods. An operator who says "check the Passengers collection" has not internalized Atom 3.5.
